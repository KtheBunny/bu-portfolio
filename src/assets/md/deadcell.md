# 分析 Dead Cells 並嘗試在 Minecraft 中重現 Roguelite × Metroidvania 的關卡生成設計

## 前言

Dead Cells 是我非常喜歡的一款作品，它成功融合了 Roguelite 與 Metroidvania 兩種原本存在一些衝突的玩法。

我希望嘗試利用 Minecraft 原生內容如 Command Block、Structure Black 等重現 Dead Cells 的核心體驗，而不是單純複製地圖或戰鬥，因此我的研究重點放在：

- Dead Cells 如何融合 Roguelite 與 Metroidvania
- Dead Cells 的隨機地城的生成邏輯
- Minecraft 是否能利用指令完成類似系統

本文將介紹我分析後的理解，以及目前實作成果與遇到的技術挑戰。

---

## Dead Cells 的核心玩法分析

Dead Cells 是一款融合 Roguelite 與 Metroidvania 的動作遊戲。其核心玩法，我認為主要有以下三個系統。

### 隨機生成的關卡

每次開始新的遊戲時，關卡都會重新生成。Dead Cells 並非完全隨機地產生整張地圖，而是將大量預先設計好的房間（Room Templates）依照特定規則重新組合，因此玩家每次探索的路線、敵人分布、商店、寶箱與特殊房間的位置都會有所不同。這種設計既能維持每次遊玩的新鮮感，又能確保關卡品質與探索節奏。

### 死亡重來，但保留永久成長

玩家在探索過程中可收集細胞（Cells），並在每個關卡之間的安全區域交給收藏家（The Collector），用來解鎖永久內容，例如提升初始血瓶數量、增加可攜帶金錢上限、解鎖新武器與技能藍圖等，讓角色可用內容越來越多。

角色死亡後，當前攜帶的裝備、金錢與未花費的細胞都會失去，玩家必須重新開始新的冒險；然而，已解鎖的永久內容會保留下來，使玩家能逐步擴充可取得的裝備池與初始能力。這種死亡重來但保留部分進度的設計，也是 Roguelite 與傳統 Roguelike 最主要的差異之一。

### 多樣化的裝備效果與 Build 組合

Dead Cells 的戰鬥深度很大程度來自於裝備之間的搭配。

每件武器除了基本攻擊方式外，還可能隨機附帶不同詞綴（Affixes），例如造成流血、中毒、冰凍、燃燒、電擊等異常狀態，或是提升對特定狀態敵人的傷害。不同效果之間還能形成聯動，例如：

- 對沾有機油的敵人施加燃燒可造成更高傷害。
- 冰凍效果搭配水域環境可延長控制時間。
- 部分武器會對流血、中毒或燃燒中的敵人造成額外傷害。

此外，每局遊戲取得的武器、技能與變異（Mutations）皆具有隨機性，因此玩家無法事先規劃完整的配裝，而是必須依據當前取得的裝備，快速分析其效果並組成適合本局的 Build。這也是 Dead Cells 每次遊玩都具有高度變化性的原因。

## Metroidvania 和 Roguelite 的融合

Merriam-Webster 將 **Metroidvania** 定義為一種結合動作與探索的遊戲類型，玩家在相互連通的世界中探索，並透過取得新能力逐步解鎖原本無法抵達的區域，形成以能力解鎖（Ability Gating）為核心的非線性探索流程。

相對地，**Roguelite** 並沒有單一的官方定義，但普遍指保留 Roguelike 的隨機生成與死亡重來機制，同時加入永久成長（Meta Progression），使玩家在每次遊玩之間仍能保留部分解鎖內容或能力。

乍看之下，這兩種類型似乎互相衝突：Metroidvania 通常依賴固定世界與能力解鎖來引導探索，而 Roguelite 則強調每次遊戲皆重新生成地圖。

然而，Dead Cells 成功將兩者結合，其關鍵並不是讓整張世界完全隨機，而是**保留固定的世界進程，並讓每個區域內部採用受規則限制的程序生成**。

### 能力所「解鎖」的內容

Dead Cells 中永久取得的符文（Runes）會解鎖新的探索能力，例如生成藤蔓、踩碎地面或使用傳送石碑等。

這些能力並不會改變關卡生成方式，而是增加玩家在探索時可互動的內容。例如：

- 通往隱藏寶箱或挑戰房間。
- 商店或劇情房間。
- 通往全新生態區域（Biome）的出口。

因此，能力解鎖的並不是固定的一條路，而是**讓之後每次生成的地圖，都有機會出現更多可探索的內容**。

例如，玩家取得藤蔓符文後，之後生成的地圖中便可能出現需要生成藤蔓才能到達的平台；取得牧羊符文（Ram Rune）後，也會開始出現可破壞的地板，進一步拓展可探索的區域。

### 不完全隨機的隨機地圖

Dead Cells 的關卡並非完全隨機，而是透過模組化（Modular）的方式生成。

玩家雖然無法預測房間的排列方式，但每個生態區域（Biome）都具有固定的設計規則。例如，在「有罪者的大道（Promenade of the Condemned）」中，玩家一定會遇到需要藤蔓符文才能繼續前進的地形，也一定會有出口、商店、獎勵房間等重要節點，只是它們在地圖中的實際位置每次都不同。

從系統設計角度來看，可以理解為每個區域都擁有一個預先設計好的房間池（Room Pool），生成器依照一定規則抽取房間進行拼接，例如：

- 固定數量的主要路徑房間。
- 若干岔路。
- 一定數量的獎勵房、商店與挑戰房。
- 固定位置的入口與出口。

因此，玩家始終擁有明確的探索目標，但到達目標的路徑卻會因地圖生成而有所不同，使 Metroidvania 的探索節奏與 Roguelite 的重玩價值得以同時成立。

## Minecraft 重現的目標

我的目標並非完整重現 Dead Cells 的所有系統，而是嘗試僅利用 Minecraft 原生提供的功能（如 Command Block、Structure Block、Scoreboard 等），實作出接近 Dead Cells 的程序式關卡生成。

在完成關卡生成系統後，再以此作為基礎，設計更符合 Minecraft 三維空間特色的探索與戰鬥玩法，逐步建立完整的 Game Loop。

目前主要實作重點包括：

- 建立房間模板（Room Templates）。
- 依照規則隨機選擇房間並生成。
- 讓房間依序連接形成完整關卡。
- 控制房間方向與連接方式，使地圖具有探索性。

在研究 Dead Cells 的關卡生成方式時，我參考了開發者 Deepnight 發表的技術文章，以及其他開發者針對其生成方式的分析。

可以發現，他們在正式生成地圖之前，都會先建立一張抽象化的**關卡流程圖（Level Graph）**，再依照流程圖中的節點決定每個房間應該生成的位置與種類，而不是直接隨機拼接房間。

這種做法讓隨機生成仍然受到設計者控制，使關卡在維持變化性的同時，也能保有探索節奏與遊戲體驗。

### 目前的實作方式

由於 Minecraft 並沒有提供程序生成相關的工具，因此我必須利用原版功能自行建立一套生成流程。

整個系統主要由 **Structure Block、Command Block、Armor Stand 與 Scoreboard** 四個部分組成。

其中：

- **Structure Block**：負責儲存與載入房間模板。
- **Armor Stand**：作為生成節點（Generation Node），代表和觸發目前需要生成下一個房間的位置，以及是讓系統保持串行的關鍵。
- **Scoreboard**：保存生成狀態，例如剩餘岔路數量、目前生成階段等資訊。
- **Command Block**：負責執行整個生成流程。

為了先驗證生成邏輯，目前僅建立三種房間模板，避免除錯成本過高：

- 藍色：一般走廊
- 橘色：岔路
- 紫色：死路

同時，每個房間皆預先建立四個方向的 Structure，以簡化旋轉時的判定流程。

![](../article/deadcell/4way.png)

另外，目前關卡架構也刻意保持簡單，使重點集中在生成演算法本身，而非房間內容。

![](../article/deadcell/graph.png)

### 生成流程

當玩家開始生成地圖時，系統首先會初始化 Scoreboard，並設定本次地圖預計生成的岔路數量，目前固定為 **2**。

這個數值同時也是限制關卡規模的重要參數，可避免地圖無限制擴張。

生成流程如下：

起始房間生成後，會初始化 Scoreboard，並設定本次地圖預計生成的岔路數量，目前固定為 2 ，這個數值同時也是限制關卡規模的重要參數，可避免地圖無限制擴張。完成後會建立第一個藍色走廊模板。

![](../article/deadcell/start.png)

藍色走廊會隨機決定：- 繼續生成走廊 - 或生成岔路。

![](../article/deadcell/blue.png)

當生成橘色岔路時：- 隨機選擇一個方向作為主路徑。- 另外兩個方向僅允許生成死路。- Scoreboard 中的剩餘岔路數減一。

![](../article/deadcell/orange.png)

若剩餘岔路數仍大於 0，則持續重複上述流程。

當岔路數歸零後，最後一個岔路會於隨機方向生成終點房間，並開始清除生成過程中建立的 Armor Stand、Command Block 等暫時物件。

### 目前遇到的技術問題

目前系統已能完成基本的房間程序生成，但仍存在幾項限制。

首先，由於 Minecraft 的 Command Block 皆依照 Game Tick 執行，若同時啟動大量 Structure Block 或 Command Block，容易造成伺服器短時間負載過高，甚至出現卡頓。因此目前系統採用**串行（Sequential）**生成，每次僅允許一個房間模板執行生成流程，避免多個節點同時運作。

然而，若未來增加更多房間種類、更複雜的生成條件或更大的地圖，此做法將使生成時間顯著增加，因此後續仍需要重新設計生成流程，以提升效率。

第二，目前大量依賴 Command Block 與 Armor Stand 作為系統運作媒介，會增加世界中的實體數量，也提高維護成本。未來希望將系統逐步移植至 Datapack，以減少 Command Block 的使用，提升可維護性與執行效率。

最後，也是目前最大的限制，是房間占用判定尚未完成。

目前只能透過：

- 固定房間尺寸
- 限制轉向次數
- 控制岔路數量

降低房間互相重疊的機率。

未來預計建立房間座標資料，並於生成前進行碰撞檢查，若判定生成位置已被占用，則重新選擇房間或重新規劃路徑，使生成器能支援更多樣的房間尺寸、分支與關卡規模。

### 長遠目標和心得

當程序生成系統穩定後，我希望將其發展成更符合 Minecraft 三維空間特色的探索玩法，而非僅複製 Dead Cells。例如：

- 更多房間模板種類
- 不同尺寸的房間
- 多樓層（Y 軸）連接

而這次實作讓我最大的收穫，是理解程序生成並不是單純的「隨機」，而是在於如何透過各種限制條件，讓每一次生成既保持變化性，又能維持關卡品質與探索節奏。

以目前系統為例，光是避免房間重疊、控制分支數量、安排生成順序等問題，就需要設計大量額外的規則，而這些都還只是最基本的生成需求。也因此，我更能理解 Dead Cells 團隊為何會先設計完整的關卡流程，再讓隨機生成建立在既定規則之上，而不是完全依賴隨機性。
